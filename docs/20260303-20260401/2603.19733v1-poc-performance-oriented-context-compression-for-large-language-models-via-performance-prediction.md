## 摘要
虽然上下文压缩可以通过缩短上下文来缓解大语言模型（LLMs）日益增长的推理成本，但现有指定目标压缩率或长度的方法面临着不可预测的性能下降，阻碍了其可靠部署。我们提出了一种向面向性能的上下文压缩（PoC）的范式转变，开发者在此模式下指定可接受的性能底线，而非压缩率。PoC 采用轻量级性能预测器，在引导现成压缩器之前，自动寻找满足该约束的最激进压缩率。我们设计并比较了两种预测器变体：一种简单的上下文无关预测器，以及一种考虑输入固有可压缩性的更复杂的上下文感知预测器。在问答和摘要基准测试中，上下文感知预测器始终比上下文无关预测器获得更低的性能预测误差，而由此产生的上下文感知 PoC 则实现了更优的整体性能。我们的工作为大语言模型上下文压缩的更可靠、高效且感知性能的部署铺平了道路。

## Abstract
While context compression can mitigate the growing inference costs of Large Language Models (LLMs) by shortening contexts, existing methods that specify a target compression ratio or length suffer from unpredictable performance degradation, hindering their reliable deployment. We introduce a paradigm shift to Performance-oriented Context Compression (PoC), where developers specify an acceptable performance floor instead of a compression ratio. PoC employs a lightweight performance predictor to automatically find the most aggressive compression ratio that satisfies this constraint before steering an off-the-shelf compressor. We design and compare two predictor variants: a simple context-agnostic predictor and a more sophisticated context-aware one that considers the input's inherent compressibility. On both question-answering and summarization benchmarks, the context-aware predictor consistently achieves lower performance prediction error than the context-agnostic predictor, while the resulting context-aware PoC attains a superior overall performance. Our work paves the way for a more reliable, efficient, and performance-aware deployment of context compression for LLMs.

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）处理长文本的需求增加，推理成本和延迟显著上升。现有的上下文压缩技术大多属于**预算导向（Budget-oriented Context Compression, BoC）**，即用户需预设固定的压缩率或长度。然而，压缩率与任务性能之间的关系高度依赖于输入内容（如信息密度不同），导致性能下降难以预测且不可控。这使得开发者在部署时面临两难：压缩太狠会损害效果，压缩太保守则无法充分提升效率。
本文提出了**面向性能的上下文压缩（Performance-oriented Context Compression, PoC）**，将控制权从“压缩预算”转移到“任务性能”，允许用户指定一个性能底线（Performance Floor），由系统自动寻找满足该约束的最优压缩比例。

#### 2. 方法论：核心思想与关键技术
PoC 框架的核心思想是在执行压缩前，先通过一个轻量级的**性能预测器**估计不同压缩率下的性能表现。

*   **核心流程**：
    1.  **性能预测**：输入上下文 $C$ 和候选压缩率序列，预测对应的性能保留程度（Performance Retention）。
    2.  **寻找压缩率**：基于预测的性能-压缩曲线，搜索满足用户性能底线 $p^*$ 的最小压缩率 $r^*$（即最激进的压缩）。
    3.  **执行压缩**：调用现成的压缩器（如 LLMLingua2）按 $r^*$ 压缩上下文，最后交给 Reader LLM 生成答案。
*   **关键技术细节**：
    *   **性能保留度定义**：将原始分数归一化为 0 到 1 之间的数值，以便跨任务统一建模。

#### 6. 主要结论与发现
*   **预测精度**：上下文感知预测器在所有任务上的 PPE 均显著低于基准（QA 任务降低 1.9%-12.8%，摘要任务降低达 21.7%-33.9%）。
*   **性能提升**：在 TriviaQA 上提升最为显著（F1@R 从 49.2 升至 67.5），原因是该数据集噪声多，适度压缩反而能起到“去噪”作用提升效果。
*   **任务差异**：QA 任务的性能曲线往往呈“悬崖式”掉落（关键信息丢失即失败），而摘要任务的性能随压缩率下降较为平缓，因此摘要任务的预测误差更小。
*   **效率**：PoC 带来的额外开销极低（约 29ms/512 tokens），但通过压缩 32k 文本可实现高达 14.5x 至 36x 的端到端推理加速。

#### 7. 优点
*   **范式创新**：从“给多少钱办多少事（BoC）”转变为“办成这样要多少钱（PoC）”，极大提升了压缩技术在生产环境中的可靠性。
*   **轻量高效**：预测器设计精巧，支持并行预测，且通过剪枝保证了极低的推理延迟。
*   **通用性强**：作为一个元框架（Meta-framework），它可以适配任何现成的硬压缩器（Hard Compressor）。

#### 8. 不足与局限
*   **依赖底层压缩器**：PoC 的性能上限受限于所使用的压缩器（如 LLMLingua2），它只能优化压缩率的选择，无法提升压缩算法本身的质量。
*   **任务覆盖**：虽然涵盖了 QA 和摘要，但对于代码生成、多轮对话等更复杂的长文本任务尚未验证。
*   **黑盒模型限制**：虽然文中提到支持黑盒模型，但训练预测器仍需要获取 Reader LLM 在不同压缩率下的反馈数据，这在某些闭源 API 场景下成本较高。

## 摘要
大语言模型（LLM）在通用编程语言中展现出强大的代码生成能力，但在底层嵌入式系统编程等专业领域仍存在局限性。该领域涉及硬件寄存器操作、厂商特定的 SDK、实时操作系统 API 以及硬件抽象层，而这些内容在标准预训练语料库中代表性不足。我们推出了 H2LooP Spark Preview，这是一个持续预训练（CPT）流水线，它在 8 台 NVIDIA H100 GPU 上使用带有秩稳定缩放（rank-stabilized scaling）的 BF16 LoRA，将完全开源的语言模型 OLMo-3-7B 适配到嵌入式系统领域。我们的训练语料库由“仓库-数据手册”对构建而成，涵盖了来自 117 家制造商的 1000 亿 token 的原始嵌入式系统数据，并采用了 SpecMap（Nipane 等人，2026）中提出的分层数据手册到代码映射方法进行处理。最终生成的精选数据集切分包含跨 13 个嵌入式领域的 235 亿 token。使用高秩 LoRA（r=512）进行的持续预训练取得了显著收益，使域内困惑度降低了 70.4%，留出仓库困惑度降低了 66.1%。在涵盖 13 个嵌入式领域的生成式代码补全基准测试中，我们的 7B 模型在 8 个类别的 token 准确率上超过了 Claude Opus 4.6 和 Qwen3-Coder-30B，这表明有针对性的持续预训练能让较小的开源权重模型在专业技术任务上与前沿系统相媲美。我们在 Huggingface 上发布了生产级训练检查点，作为开源成果。

## Abstract
Large language models (LLMs) demonstrate strong code generation abilities in general-purpose programming languages but remain limited in specialized domains such as low-level embedded systems programming. This domain involves hardware register manipulation, vendor-specific SDKs, real-time operating system APIs, and hardware abstraction layers that are underrepresented in standard pretraining corpora. We introduce H2LooP Spark Preview, a continual pretraining (CPT) pipeline that adapts the OLMo-3-7B-a fully open language model to the embedded systems domain using BF16 LoRA with rank-stabilized scaling on 8 NVIDIA H100 GPUs. Our training corpus is constructed from repository-datasheet pairs covering 100B tokens of raw embedded systems data across 117 manufacturers, processed using the hierarchical datasheet-to-code mapping approach proposed in SpecMap (Nipane et al., 2026). The resulting curated dataset split contains 23.5B tokens across 13 embedded domains. Continual pretraining with high-rank LoRA (r=512) yields substantial gains, reducing in-domain perplexity by 70.4% and held-out repository perplexity by 66.1%. On generative code completion benchmarks spanning 13 embedded domains, our 7B model outperforms Claude Opus 4.6 and Qwen3-Coder-30B on 8 categories in token accuracy, showing that targeted continual pretraining enables smaller open-weight models to rival frontier systems on specialized technical tasks. We release the production training checkpoint on Huggingface as an open-source artifact.

#### 1. 核心问题与研究动机
*   **核心问题**：通用大语言模型（LLMs）在底层嵌入式系统编程（Low-Level Embedded Systems Code）方面存在显著局限。
*   **研究背景**：嵌入式领域具有极高的硬件特异性，涉及特定厂商的寄存器映射、SDK 模式、实时操作系统（RTOS）API 和硬件抽象层（HAL）。这些数据在通用的预训练语料库中严重匮乏，导致现有模型（如 Claude 4.6, Qwen3-30B）在生成此类代码时经常出现寄存器名称幻觉、初始化序列错误及违反硬件时序约束等问题。

#### 2. 方法论与关键技术
*   **核心思想**：通过**持续预训练（Continual Pretraining, CPT）**，将领域特定的硬件知识注入开源基础模型（OLMo-3-7B），使其在不丢失通用能力的前提下掌握底层嵌入式逻辑。
*   **关键技术细节**：
    *   **数据构建（SpecMap）**：利用分层映射方法，将硬件数据手册（Datasheet）与对应的 GitHub 代码仓库进行关联，确保模型学习到“规格说明到代码实现”的语义联系。
    *   **高秩 LoRA 适配**：采用 **BF16 LoRA** 结合**秩稳定缩放（RSLoRA）**技术。公式上，RSLoRA 将缩放因子设为 $\alpha/\sqrt{r}$（而非标准 LoRA 的 $\alpha/r$），以支持在极高秩（$r=512$）下的稳定训练。
    *   **全模块覆盖**：LoRA 适配器不仅应用于注意力层，还覆盖了 MLP 层和嵌入层（Embedding/LM-head），总训练参数量约 8.39 亿（占原模型的 12%）。
    *   **双优化器策略**：对嵌入层使用较低的学习率（主学习率的 0.5 倍），以保护预训练的词表表示。

*   **对比方法**：

#### 6. 主要结论与发现
*   **性能飞跃**：Spark Preview 7B 在 13 个类别中的 8 个类别上，Token 准确率超越了参数量大得多的 Claude Opus 4.6 和 Qwen3-30B。
*   **困惑度大幅下降**：域内困惑度降低了 70.4%（4.06 → 1.20），留出仓库降低了 66.1%，证明了极强的领域适配效果。
*   **高秩必要性**：实验证明 $r=512$ 的高秩 LoRA 对领域知识注入至关重要，且必须配合保守的学习率（$1.5 \times 10^{-5}$）以防止训练崩溃。

*   **开源贡献**：发布了生产级检查点，填补了开源社区在底层嵌入式 AI 模型上的空白。

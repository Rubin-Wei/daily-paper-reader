## 摘要
最近的对话记忆系统在数据摄入阶段大量投入基于大语言模型（LLM）的结构化工作，并在查询阶段使用学习到的检索策略。我们证明这两者都不是必需的。SmartSearch 使用完全确定性的流水线从原始、非结构化的对话历史中进行检索：包括用于召回的 NER 加权子字符串匹配、用于多跳扩展的基于规则的实体发现，以及一个 CrossEncoder+ColBERT 排序融合阶段（这是唯一的学习组件），在 CPU 上运行时间约为 650 毫秒。在两个基准测试上的 Oracle 分析确定了一个编译瓶颈：检索召回率达到了 98.6%，但如果没有智能排序，只有 22.5% 的黄金证据能在截断至 Token 预算后存留。通过分数自适应截断且无需针对每个数据集进行微调，SmartSearch 在 LoCoMo 上达到了 93.5%，在 LongMemEval-S 上达到了 88.4%，在相同评估协议下，在两个基准测试中均超过了所有已知的记忆系统，同时使用的 Token 数量比全上下文基准少 8.5 倍。

## Abstract
Recent conversational memory systems invest heavily in LLM-based structuring at ingestion time and learned retrieval policies at query time. We show that neither is necessary. SmartSearch retrieves from raw, unstructured conversation history using a fully deterministic pipeline: NER-weighted substring matching for recall, rule-based entity discovery for multi-hop expansion, and a CrossEncoder+ColBERT rank fusion stage -- the only learned component -- running on CPU in ~650ms. Oracle analysis on two benchmarks identifies a compilation bottleneck: retrieval recall reaches 98.6%, but without intelligent ranking only 22.5% of gold evidence survives truncation to the token budget. With score-adaptive truncation and no per-dataset tuning, SmartSearch achieves 93.5% on LoCoMo and 88.4% on LongMemEval-S, exceeding all known memory systems under the same evaluation protocol on both benchmarks while using 8.5x fewer tokens than full-context baselines.

### 2. 方法论：SmartSearch 流水线
SmartSearch 采用一个完全确定性的检索流程，仅在最后的重排序阶段使用轻量级模型：
*   **查询解析（Parse）**：使用 SpaCy 进行 NER（命名实体识别）和 POS（词性标注）。根据语言学特征为术语加权：专有名词(3.0) > 名词(2.0) > 动词(1.0)，实体额外+1.0。这取代了传统的 BM25 统计加权。
*   **确定性检索（Retrieve）**：
    *   **子串匹配（grep）**：基于加权术语进行精确匹配，处理了 98.9% 的案例。
    *   **多跳扩展（Multi-hop Expansion）**：通过规则在检索到的片段中发现新实体，自动触发后续搜索，无需 LLM 介入。
    *   **语义回退**：仅在子串匹配失败时（约 1%）使用 ColBERT 向量检索。
*   **排序融合（Rank）**：这是唯一的学习组件。将 **CrossEncoder** (mxbai-rerank-large) 与 **ColBERT** 的评分通过 **RRF (Reciprocal Rank Fusion)** 融合。
*   **自适应截断（Truncate）**：提出**分数自适应截断（Score-adaptive Truncation）**，根据重排序分数动态决定提供给 LLM 的上下文长度，而非固定 Token 预算。

*   **极低成本**：无需昂贵的 LLM 预处理，无需 GPU 推理检索，显著降低了系统总成本（TCO）。

## 摘要
大语言模型（LLMs）现在已经能够解决真实的 GitHub 问题。然而，当前的方法对代码上下文进行了过度近似，并面临两个复合问题：处理海量输入的高昂成本，以及由于噪声充斥上下文窗口并分散模型对错误修复信号的注意力而导致的低效。现有的压缩技术未能解决这一矛盾：通用压缩器会损害代码的语义完整性，而特定于代码的工具则缺乏对代码结构和任务上下文的感知，无法保留必要的补丁要素。为了解决这一问题，我们提出了一个由两个组件组成的新型框架。首先是 Oracle 引导的代码蒸馏（OCD），这是一种结合了遗传算法和增量调试（delta debugging）的上下文蒸馏算法，旨在系统地将代码上下文缩减为最小充分子序列——仅保留成功修复所需的要素。我们利用这些蒸馏数据来微调 SWEzze，这是一个轻量级模型，能够在推理时学习压缩代码上下文，在过滤噪声和对抗干扰的同时保留修复要素。在 SWE-bench Verified 基准测试中，通过对三种前沿 LLM 的评估，SWEzze 在不同模型上保持了约 6 倍的稳定压缩率，相对于未压缩设置减少了 51.8%-71.3% 的总 Token 预算，将问题解决率提高了 5.0%-9.2%，并且与最先进的上下文压缩基准相比，在有效性、压缩比和延迟之间实现了最佳的整体平衡。

## Abstract
Large Language Models (LLMs) are now capable of resolving real-world GitHub issues. However, current approaches overapproximate the code context and suffer from two compounding problems: the prohibitive cost of processing massive inputs, and low effectiveness as noise floods the context window and distracts the model from the bug-fixing signal. Existing compression techniques fail to resolve this tension: generic compressors compromise the semantic integrity of code, while code-specific tools lack awareness of code structure and task context to preserve essential patch ingredients. To address this, we propose a novel framework consisting of two components. First, Oracle-guided Code Distillation (OCD), a context distillation algorithm that combines genetic search and delta debugging to systematically reduce code contexts to their minimal sufficient subsequence - retaining only the ingredients required for a successful fix. We use this distilled data to fine-tune SWEzze, a lightweight model that learns to compress code context at inference time, filtering noise and combating distraction while preserving fix ingredients. Evaluated on SWE-bench Verified across three frontier LLMs, SWEzze maintains a stable compression rate of about 6 times across models, reduces the total token budget by 51.8%-71.3% relative to the uncompressed setting, improves issue resolution rates by 5.0%-9.2%, and delivers the best overall balance among effectiveness, compression ratio, and latency compared with state-of-the-art context compression baselines.

### 论文总结：基于大语言模型的问题修复中的代码上下文压缩

#### 1. 核心问题与整体含义（研究动机和背景）
在利用大语言模型（LLM）自动修复 GitHub 真实问题的任务中，构建有效的代码上下文至关重要。然而，当前方法普遍存在**上下文过度近似（Overapproximation）**的问题：
*   **高昂成本**：为了不遗漏关键依赖，检索工具往往返回大量文件，导致推理 Token 成本激增。
*   **噪声干扰**：冗余代码会充斥上下文窗口，分散模型注意力（Lost-in-the-middle 现象），降低修复成功率。
*   **现有压缩技术的缺陷**：通用压缩器（如 LLMLingua）会破坏代码语义结构；而现有的代码压缩器（如 LongCodeZip）多基于词法相似度，容易剔除那些虽然与 Bug 描述不相似但对构造补丁至关重要的“修复成分（Fix Ingredients）”。

#### 2. 方法论：核心思想与关键技术
本文提出了一个结合“离线蒸馏”与“在线压缩”的框架，核心由 OCD 算法和 SWEzze 模型组成：

*   **Oracle 引导的代码蒸馏 (OCD)**：
    *   **目标**：寻找“最小充分上下文（Minimal Sufficient Context, MSC）”，即移除其中任何一个片段都会导致修复失败的代码子集。
    *   **两阶段搜索**：
        1.  **遗传算法 (GA)**：在指数级的代码组合空间中搜索，利用优先级评分（基于补丁重合度、测试覆盖率、符号重合度）引导，找到能使模型生成正确补丁的初始子集。
        2.  **分层增量调试 (HDD)**：在文件、函数、块三个层级上应用 `ddmin` 算法，系统性地剔除冗余单元，确保最终上下文的 1-最小性（1-minimality）。
*   **SWEzze 压缩模型**：
    *   **模型架构**：基于 `Qwen3-Reranker-0.6B` 轻量级交叉编码器，使用 LoRA 进行参数高效微调。
    *   **训练信号**：使用 OCD 产生的 MSC 作为金标准，学习将（问题描述 + 故障位置 + 原始上下文）映射到关键代码段。
    *   **关键机制**：引入**角色感知样本加权（Role-aware weighting）**，针对 Schema 定义、调用链等不同语义角色赋予不同权重，解决正负样本极度不平衡的问题（相关代码段仅占约 8.4%）。
*   **推理流程**：在推理时，SWEzze 对所有代码段评分，并根据预设的 Token 预算进行贪婪选择，组装压缩后的上下文。

*   **实验规模**：在 500 个标准 Benchmark 实例上进行了跨 3 种不同架构 LLM 的全量评估，实验组数充足。

#### 6. 主要结论与发现
*   **性能提升**：SWEzze 将问题解决率提升了 **5.0%–9.2%**（相比未压缩设置），证明了“去噪”对修复任务的正面作用。
*   **成本削减**：实现了约 **6 倍** 的稳定压缩率，减少了 **51.8%–71.3%** 的总 Token 预算。
*   **泛化性**：虽然训练数据由 Qwen3-Coder-30b 产生，但压缩结果在 GPT-5.2 和 DeepSeek-V3.2 上同样有效，说明“修复成分”具有模型无关的语义通用性。
*   **关键发现**：Schema 定义（数据类、类型定义）的修复相关性最高（39.2%），而通用工具类代码往往是主要的噪声来源。

#### 7. 优点
*   **任务导向**：不同于以往基于相似度的压缩，该方法直接以“能否修复成功”作为蒸馏目标，抓住了问题的核心。
*   **结构保护**：通过分层分解和占位符（Placeholder）机制，在大幅删减代码的同时保留了语法结构的完整性。
*   **效率平衡**：在保持高压缩质量的同时，推理延迟远低于基于 Perplexity 的复杂搜索方法。

#### 8. 不足与局限
*   **语法破坏风险**：尽管有结构保护，但极度压缩（如 Block 级别）仍可能导致代码片段过于破碎，偶尔会误导 LLM 产生错误的逻辑假设。
*   **语义判别精度**：在处理具有多个相似实现（如重写函数）的大型项目时，模型有时难以精确区分哪个才是真正的修复目标。
*   **依赖 Oracle**：训练数据的获取依赖于高质量的测试套件和已有的正确补丁，对于缺乏测试覆盖的项目，难以生成高质量的训练数据。

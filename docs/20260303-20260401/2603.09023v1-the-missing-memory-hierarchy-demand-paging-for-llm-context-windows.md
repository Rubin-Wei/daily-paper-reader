## 摘要
大语言模型的上下文窗口并非内存。它是 L1 缓存：一种小容量、高速且昂贵的资源，而该领域却将其视为整个内存系统。这里没有 L2，没有虚拟内存，也没有分页。在会话的整个生命周期中，每个工具定义、每个系统提示词以及每个过时的工具结果都会占用上下文。结果是可衡量的：在 857 个生产会话和 445 万个有效输入 token 中，21.8% 是结构性浪费。我们提出了 Pichay，一个针对 LLM 上下文窗口的请求分页系统。Pichay 作为客户端和推理 API 之间的透明代理实现，它介入消息流以驱逐陈旧内容，在模型重新请求已驱逐材料时检测缺页异常，并固定由缺页历史记录识别的工作集页面。在 140 万次模拟驱逐的离线重放中，缺页率为 0.0254%。在超过 681 轮的实际生产部署中，该系统将上下文消耗降低了高达 93%（从 5,038KB 降至 339KB）；在极端的持续压力下，系统仍能运行，但表现出预期的抖动病态，即已驱逐内容的重复调入。关键观察在于，该领域面临的问题（如上下文限制、注意力退化、成本扩展、跨会话状态丢失）本质上是披着不同外衣的虚拟内存问题。解决方案已经存在：工作集理论（Denning, 1968）、请求分页、缺页驱动的替换策略，以及具有多个驱逐管理层级的内存层级结构。我们描述了 LLM 系统完整内存层级（从 L1 到持久存储）的架构，报告了在生产中部署的前三个层级（L1 驱逐、L2 缺页驱动固定、L3 模型发起的对话压缩），并将跨会话内存确定为剩余的前沿领域。

## Abstract
The context window of a large language model is not memory. It is L1 cache: a small, fast, expensive resource that the field treats as the entire memory system. There is no L2, no virtual memory, no paging. Every tool definition, every system prompt, and every stale tool result occupies context for the lifetime of the session. The result is measurable: across 857 production sessions and 4.45 million effective input tokens, 21.8% is structural waste.   We present Pichay, a demand paging system for LLM context windows. Implemented as a transparent proxy between client and inference API, Pichay interposes on the message stream to evict stale content, detect page faults when the model re-requests evicted material, and pin working-set pages identified by fault history. In offline replay across 1.4 million simulated evictions, the fault rate is 0.0254%. In live production deployment over 681turns, the system reduces context consumption by up to 93% (5,038KB to 339KB); under extreme sustained pressure, the system remains operational but exhibits the expected thrashing pathology, with repeated fault-in of evicted content.   The key observation is that the problems the field faces, such as context limits, attention degradation, cost scaling, lost state across sessions, are virtual memory problems wearing different clothes. The solutions exist: working set theory (Denning, 1968), demand paging, fault-driven replacement policies, and memory hierarchies with multiple eviction-managed levels. We describe the architecture of a full memory hierarchy for LLM systems (L1 through persistent storage), report on the first three levels deployed in production use (L1 eviction, L2 fault-driven pinning, L3 model-initiated conversation compaction), and identify cross-session memory as the remaining frontier.

### 论文总结：缺失的内存层级——LLM 上下文窗口的请求分页

#### 1. 核心问题与整体含义
论文指出，当前大语言模型（LLM）领域对“上下文窗口”（Context Window）的使用存在严重的架构误区：开发者将其视为**物理内存**（全量堆叠），而其本质更接近**L1 缓存**（昂贵、高速、容量有限）。
*   **研究动机**：在 Agent（智能体）应用中，工具定义、系统提示词和历史结果在每一轮 API 调用中都会被全量发送，导致严重的“结构性浪费”。
*   **核心背景**：随着会话增长，上下文的二次方计算成本（Attention 机制）导致成本激增和性能下降。目前的行业做法是单纯增加窗口大小（类似增加 RAM），而非引入成熟的虚拟内存管理机制。

#### 2. 方法论：Pichay 系统
作者提出了 **Pichay**，一个位于客户端与推理 API 之间的透明 HTTP 代理系统，实现了类似操作系统的虚拟内存管理。
*   **请求分页（Demand Paging）**：将上下文内容分为“垃圾回收”（不可重读的临时输出）和“分页”（可重读的文件内容）。
*   **驱逐策略（Eviction Policy）**：采用 FIFO（先进先出）算法，将超过阈值的陈旧内容移出上下文，替换为微小的“检索句柄”（Retrieval Handles/Tombstones）。
*   **缺页中断检测（Page Fault Detection）**：当模型通过工具调用重新请求已被驱逐的内容时，系统识别为“缺页”，并从后台存储中重新调入。
*   **缺页驱动固定（Fault-Driven Pinning）**：如果某内容被驱逐后触发了缺页，系统会将其“固定”在 L2 工作集中，防止再次驱逐。
*   **协作式内存管理**：
    *   **幻影工具（Phantom Tools）**：注入模型可见但框架不可见的工具，允许模型主动释放内存（`memory_release`）或请求调入（`memory_fault`）。
    *   **清理标签（Cleanup Tags）**：模型可在输出中嵌入指令，主动要求压缩（`collapse`）或总结历史对话。

#### 3. 实验设计
*   **数据集/场景**：基于 **Claude Code**（Anthropic 的 AI 编码助手）的真实生产数据，包含 857 个会话、54,170 次 API 调用，涉及 44.5 亿个有效 token。
*   **Benchmark**：以原始全量上下文发送为 Baseline。
*   **对比方法**：
    1.  **Baseline**：无干预。
    2.  **Trimmed**：仅进行工具定义精简和去重。
    3.  **Compact+Trim**：在精简基础上加入请求分页（Pichay 核心功能）。
*   **验证方式**：包括离线重放（140 万次模拟驱逐）、生产环境实测（681 轮超长会话）以及 LLM 盲测质量评估。

#### 4. 资源与算力
*   **模型使用**：主要使用 Anthropic 的 **Claude 3 Opus**、**Sonnet 3.5** 和 **Haiku 4.5**。
*   **算力说明**：论文未提及具体的 GPU 训练算力，因为 Pichay 是一个**推理侧代理系统**，不需要重新训练模型。其核心开销在于代理层的 JSON 解析和元数据管理，对延迟影响极小。

#### 5. 实验数量与充分性
*   **实验规模**：分析了 857 个生产会话，样本量较大且具有代表性。
*   **消融实验**：通过对比 Trimmed 和 Compact 模式，清晰地分离了“静态精简”与“动态分页”的贡献。
*   **客观性**：引入了由三个 LLM 组成的评审团进行双盲质量评估（18 个会话，54 个判定），验证了上下文压缩后模型能力的非劣效性。
*   **充分性**：实验覆盖了从短会话到极端长会话（681 轮）的多种场景，并记录了“抖动”（Thrashing）等病态情况，表现出较高的客观性。

#### 6. 主要结论与发现
*   **浪费严重**：生产环境中 21.8% 的 token 是结构性浪费（未使用工具定义占 11%，重复内容占 2.2%，陈旧结果占 8.7%）。
*   **压缩率极高**：在长会话中，上下文消耗可从 5,038KB 降低至 339KB（**-93%**）。
*   **缺页率极低**：离线重放显示的缺页率仅为 **0.0254%**，证明 FIFO 策略在 LLM 场景下意外地高效。
*   **成本模型反转**：在传统 OS 中，保留内存是免费的，缺页是昂贵的；在 LLM 中，**保留 token 是昂贵的（每轮都要付钱），而缺页（重新读取一次）相对便宜**。

#### 7. 优点
*   **架构创新**：首次将经典的虚拟内存层级（L1-L4）完整映射到 LLM 上下文管理中。
*   **透明性**：作为代理运行，无需修改模型权重、客户端代码或推理 API。
*   **协作性**：提出的“幻影工具”和“清理标签”开创了模型主动参与内存管理的新范式。
*   **实战导向**：数据来源于真实生产环境，而非合成数据集，具有极强的工程参考价值。

#### 8. 不足与局限
*   **样本偏差**：语料库主要来自单一高级用户（作者本人）的编码任务，可能无法完全代表所有 AI 应用场景。
*   **抖动风险**：在极端压力下，系统会出现频繁的驱逐与调入循环（Thrashing），导致 API 成本反而上升。
*   **引用失效**：如果用户在提示词中隐式引用了已被驱逐的内容（如“根据刚才那个文件的第 10 行...”），系统可能无法识别这种前向依赖，导致模型回答失败。
*   **缓存失效成本**：频繁的结构性修改（如 L3 压缩）会导致推理侧的 Prompt Cache 失效，产生一次性的高额计算开销。

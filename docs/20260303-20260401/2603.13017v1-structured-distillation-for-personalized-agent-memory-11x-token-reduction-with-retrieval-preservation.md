## 摘要
与 AI 智能体的长期对话给用户带来了一个简单的问题：历史记录很有用，但逐字携带这些记录的成本很高。我们研究了个性化智能体记忆：将单个用户与智能体的对话历史蒸馏为一个紧凑的检索层，以便后续搜索。每次交互都被压缩为一个包含四个字段的复合对象（exchange_core、specific_context、thematic room_assignments 以及通过正则表达式提取的 files_touched）。可搜索的蒸馏文本平均每次交互仅占 38 个 token。该方法应用于来自 6 个软件工程项目的 4,182 场对话（14,340 次交互），将平均交互长度从 371 个 token 减少到 38 个 token，实现了 11 倍的压缩。我们通过 201 个面向召回的查询、涵盖 5 种纯搜索模式和 5 种跨层搜索模式的 107 种配置，以及 5 个 LLM 评分器（共 214,519 个经共识评分的查询-结果对）评估了个性化召回在压缩后是否依然有效。最佳的纯蒸馏配置达到了最佳逐字匹配 MRR 的 96%（0.717 对比 0.745）。结果取决于具体机制。在 Bonferroni 校正后，所有 20 种向量搜索配置的差异均不显著，而所有 20 种 BM25 配置的性能均显著下降（效应量 |d|=0.031-0.756）。最佳的跨层设置略微超过了最佳的纯逐字基准（MRR 0.759）。结构化蒸馏在压缩单用户智能体记忆的同时，并不会统一地牺牲检索质量。在仅需 1/11 上下文成本的情况下，数千次交互即可容纳在单个提示词中，同时逐字源码仍可用于深入查看。我们将实现代码和分析流水线作为开源软件发布。

## Abstract
Long conversations with an AI agent create a simple problem for one user: the history is useful, but carrying it verbatim is expensive. We study personalized agent memory: one user's conversation history with an agent, distilled into a compact retrieval layer for later search. Each exchange is compressed into a compound object with four fields (exchange_core, specific_context, thematic room_assignments, and regex-extracted files_touched). The searchable distilled text averages 38 tokens per exchange. Applied to 4,182 conversations (14,340 exchanges) from 6 software engineering projects, the method reduces average exchange length from 371 to 38 tokens, yielding 11x compression. We evaluate whether personalized recall survives that compression using 201 recall-oriented queries, 107 configurations spanning 5 pure and 5 cross-layer search modes, and 5 LLM graders (214,519 consensus-graded query-result pairs). The best pure distilled configuration reaches 96% of the best verbatim MRR (0.717 vs 0.745). Results are mechanism-dependent. All 20 vector search configurations remain non-significant after Bonferroni correction, while all 20 BM25 configurations degrade significantly (effect sizes |d|=0.031-0.756). The best cross-layer setup slightly exceeds the best pure verbatim baseline (MRR 0.759). Structured distillation compresses single-user agent memory without uniformly sacrificing retrieval quality. At 1/11 the context cost, thousands of exchanges fit within a single prompt while the verbatim source remains available for drill-down. We release the implementation and analysis pipeline as open-source software.

*   **研究动机**：现有的“压缩（Compaction）”方法通常是简单的摘要，缺乏结构且容易在多次迭代中丢失关键技术细节。论文旨在探索一种**结构化蒸馏**方法，为单用户建立一个极度紧凑（11倍压缩）但仍能保持高检索质量的“记忆层”，使智能体能像人类一样通过碎片化线索找回历史记忆。

### 2. 方法论：核心思想与关键技术
*   **核心思想**：将每次对话交换（Exchange）转化为一个包含四个关键字段的**结构化复合对象（Palace Object）**，并遵循“幸存词汇原则（Surviving Vocabulary Principle）”，即强制 LLM 保留原始对话中的精确术语。
*   **关键技术细节**：
    *   **Exchange 分段**：基于用户请求-助手响应的周期进行切分，过滤掉少于 100 字符的琐碎交互。
    *   **四个结构化字段**：
        1.  `exchange_core`：1-2 句总结（类似 Git Commit 消息）。
        2.  `specific_context`：一个具体的硬核技术细节（如错误码、参数名、文件路径）。
        3.  `room_assignments`：1-3 个主题标签（文件、概念或工作流类型）。
        4.  `files_touched`：通过正则提取的受影响文件路径。
    *   **两层架构**：蒸馏后的文本仅作为“路由索引”，用户最终看到的检索结果依然是原始的逐字文本。

### 3. 实验设计
*   **数据集**：来自 6 个真实软件工程项目的单用户语料库，包含 4,182 场对话、14,340 次交互。
*   **Benchmark**：201 个由人工策划的召回导向查询（分为概念型、短语型、精确术语型）。
*   **对比方法**：
    *   **5 种纯搜索模式**：逐字全文（基准）、蒸馏核心、核心+文件、核心+主题、全字段。
    *   **检索机制**：向量搜索（HNSW、Exact 相似度）和关键词搜索（BM25-Okapi、BM25-FTS）。
    *   **跨层融合**：将逐字文本的关键词搜索与蒸馏文本的向量搜索进行加权融合。

### 4. 资源与算力

*   **嵌入模型**：使用 `all-MiniLM-L6-v2`（22M 参数），选型理由是其可在单机 CPU 上运行，符合单开发者部署环境。

### 5. 实验数量与充分性
*   **实验规模**：测试了 **107 种不同的配置组合**（涵盖不同字段组合、检索算法和融合策略）。
*   **充分性**：实验设计非常详尽，不仅对比了 MRR、P@1 等硬指标，还引入了 5 个 LLM 评分器的共识机制，并针对争议样本引入了 Claude Opus 进行人工校准。
*   **客观性**：使用了 Bonferroni 校正进行统计显著性检验，并报告了效应量（Cohen's d），区分了向量搜索和 BM25 在压缩下的不同表现，结论较为客观。

### 6. 主要结论与发现
*   **极高压缩率**：实现了 **11 倍** 的 Token 缩减（平均每条记录从 371 降至 38 Token）。
*   **检索质量保留**：最佳纯蒸馏配置达到了逐字基准 MRR 的 **96%**（0.717 vs 0.745）。
*   **机制依赖性**：**向量搜索对蒸馏具有极强的鲁棒性**（性能下降不显著），而 **BM25 性能显著下降**，因为压缩移除了大量关键词。
*   **互补性**：跨层融合（逐字 BM25 + 蒸馏向量搜索）表现最好（MRR 0.759），证明蒸馏层提供了原始文本缺失的语义信号。

### 7. 优点
*   **实用性极强**：直接解决了 RAG 系统中“上下文窗口昂贵”与“长期记忆需求”的矛盾。
*   **架构优雅**：通过“索引与显示分离”的二层架构，既享受了压缩带来的低成本，又保证了用户阅读体验的真实性。
*   **幸存词汇原则**：通过约束 LLM 提取而非泛泛摘要，有效保留了技术语境下的关键“抓手”。

### 8. 不足与局限
*   **评分器一致性较低**：5 个本地 LLM 评分器之间的 Fleiss' κ 仅为 0.175（轻微一致），反映出软件工程语境下的相关性判定对小模型仍具挑战。
*   **单用户偏差**：所有数据来自单一开发者，虽然符合“个性化记忆”的主题，但其结论在不同语种、非技术领域或多用户场景下的泛化性有待验证。
*   **BM25 劣势**：如果系统强依赖传统关键词检索，该蒸馏方法会导致明显的性能损失。
*   **未评估任务性能**：论文证明了“能搜到”，但未直接证明智能体在仅持有蒸馏记忆时，完成复杂编码任务的能力是否受损。

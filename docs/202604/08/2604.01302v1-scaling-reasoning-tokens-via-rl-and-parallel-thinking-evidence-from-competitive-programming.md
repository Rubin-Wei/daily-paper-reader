## 摘要
我们研究了如何通过两种互补的方法来扩展竞赛编程的推理 Token 预算：训练时的强化学习 (RL) 和测试时的并行思考。在 RL 训练期间，我们观察到验证准确率与连续检查点生成的平均推理 Token 数量之间存在近似对数线性关系，并展示了两种改变这一训练轨迹的方法：验证 RL 预热提高了起点，而随机裁剪在观察范围内产生了更陡峭的趋势。由于在全注意力机制下，RL 过程中扩展单次生成的推理成本迅速增加，我们引入了一种多轮并行思考流水线，将 Token 预算分配到生成、验证和细化的多个线程和轮次中。我们在此流水线上对模型进行端到端训练，使训练目标与测试时的结构相匹配。以 Seed-OSS-36B 为起点，该完整系统在每个问题平均使用 760 万个 Token 的情况下，通过 16 个线程和每个线程 16 轮的配置，在 pass@1 指标上达到了基础 RL 模型 oracle pass@16 的水平，并在 AetherCode 的 456 个困难竞赛编程问题上超越了 GPT-5-high。

## Abstract
We study how to scale reasoning token budgets for competitive programming through two complementary approaches: training-time reinforcement learning (RL) and test-time parallel thinking. During RL training, we observe an approximately log-linear relationship between validation accuracy and the average number of generated reasoning tokens over successive checkpoints, and show two ways to shift this training trajectory: verification RL warmup raises the starting point, while randomized clipping produces a steeper trend in the observed regime. As scaling single-generation reasoning during RL quickly becomes expensive under full attention, we introduce a multi-round parallel thinking pipeline that distributes the token budget across threads and rounds of generation, verification, and refinement. We train the model end-to-end on this pipeline to match the training objective to the test-time structure. Starting from Seed-OSS-36B, the full system with 16 threads and 16 rounds per thread matches the underlying RL model's oracle pass@16 at pass@1 using 7.6 million tokens per problem on average, and surpasses GPT-5-high on 456 hard competitive programming problems from AetherCode.

### 1. 核心问题与整体含义
论文探讨了如何通过扩展**推理 Token（Reasoning Tokens）**的预算来提升大语言模型在复杂逻辑任务（如竞赛编程）中的表现。研究背景在于，虽然 OpenAI o1 等模型证明了强化学习（RL）能诱导更长的思维链（CoT），但单次生成的长度受限于计算成本（注意力机制的平方复杂度）。论文旨在通过训练时的 RL 优化和测试时的“并行思考”架构，突破单次生成的长度瓶颈，实现推理能力的持续扩展。

### 2. 方法论：核心思想与关键技术
论文提出了两阶段的扩展策略：

*   **训练时 RL 优化（Training-time RL Scaling）：**
    *   **对数线性趋势：** 发现验证准确率与平均推理 Token 数的对数呈线性关系。
    *   **验证 RL 预热（Verification RL Warmup）：** 在生成 RL 之前，先训练模型进行“验证”任务（判断代码正误并给出理由），这能显著提高生成 RL 的起点。
    *   **随机裁剪（Randomized Clipping, RC）：** 为了解决硬性 Token 限制导致的奖励断崖，引入随机采样长度上限，将奖励函数平滑化，从而诱导模型在更经济的预算内产生更高质量的推理，使扩展曲线更陡峭。
*   **测试时并行思考（Test-time Parallel Thinking）：**
    *   **多线程多轮次流水线：** 系统启动 $N$ 个独立线程，每个线程进行 $M$ 轮“生成-验证-精炼”循环。
    *   **自验证与精炼：** 模型利用自身训练出的验证能力对候选代码打分，若不通过则根据验证反馈进行自我修正（Self-refinement）。
    *   **端到端 RL 训练：** 为了消除训练与测试的分布差异，使用“Turn-grouped GRPO”算法对多轮对话进行端到端训练，独立校准每一轮（生成、验证、精炼）的优势函数。

### 3. 实验设计
*   **数据集：** 使用 **AetherCode**（包含 456 个来自 IOI、ICPC 等顶级赛事的难题）作为验证集。训练集包含约 10K 个专有竞赛编程题目。
*   **基准模型：** 以 **Seed-OSS-36B** 为基础模型。
*   **对比对象：** 
    *   **GPT-5-high**（OpenAI 报告中的高性能推理模式）。
    *   不同的 RL 变体（基础 GRPO vs. 随机裁剪 vs. 验证预热）。
    *   不同的测试时策略（纯并行生成 vs. 纯序列精炼 vs. 并行+序列结合）。

*   **硬件：** 使用了 **256 或 512 张 NVIDIA A100 GPU**。

### 5. 实验数量与充分性
*   **实验规模：** 论文进行了多维度的消融实验，包括验证预热的效果、随机裁剪的斜率对比、线程数 $N$ 与轮次 $M$ 的扩展性分析、以及验证次数 $V$ 对排序质量的影响。
*   **充分性：** 实验设计较为充分，不仅展示了最终性能，还通过 Oracle pass@k 曲线分析了 RL 是否真正扩展了模型的解空间（Coverage）。对比 GPT-5-high 提供了极具说服力的外部基准。

### 6. 主要结论与发现
*   **推理 Token 价值：** 准确率随推理 Token 增加而提升，且在测试时通过并行和序列结合，可扩展至单题平均 **760 万 Token**。
*   **序列优于并行：** 在相同 Token 预算下，序列化的“验证-精炼”比简单的并行采样更高效。
*   **性能突破：** 完整系统在 AetherCode 上的 pass@1 表现匹配了基础模型的 Oracle pass@16，并显著超越了 GPT-5-high。
*   **RL 的本质：** 实验证明 RL 不仅仅是浓缩 SFT 的分布，而是确实扩展了模型在复杂组合空间中发现新解的能力。

### 7. 优点
*   **系统性：** 统一了训练时和测试时的 Token 扩展视角，提供了清晰的对数线性分析框架。
*   **创新性：** 提出的“随机裁剪”是一种优雅且低成本的长度控制手段；“并行思考”架构有效规避了长序列的计算瓶颈。
*   **实战性：** 在极高难度的竞赛编程任务上取得了 SOTA 结果，证明了方法的有效性。

### 8. 不足与局限
*   **计算成本：** 测试时单题消耗 7.6M Token 的成本极高，虽然提升了性能，但在实际应用中的经济性有待商榷。
*   **验证瓶颈：** 消融实验显示，验证器的能力（而非采样次数）是目前的性能瓶颈，如果验证器判断错误，精炼过程可能会误导模型。
*   **领域局限：** 实验集中在编程领域，该领域有明确的执行反馈（Unit Tests），在缺乏自动反馈的通用推理领域（如创意写作或开放式问答）可能难以直接迁移。

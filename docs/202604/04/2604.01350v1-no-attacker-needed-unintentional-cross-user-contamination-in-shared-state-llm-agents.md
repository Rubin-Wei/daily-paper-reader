## 摘要
基于大语言模型（LLM）的智能体越来越多地在重复会话中运行，通过维护任务状态来确保连续性。在许多部署场景中，单个智能体为团队或组织内的多个用户提供服务，在不同用户身份之间复用共享知识层。这种共享持久性扩大了失效面：当智能体不考虑作用域而重新应用信息时，对一个用户局部有效的信息可能会悄无声息地降低另一个用户的输出质量。我们将这种失效模式称为无意间跨用户污染（UCC）。与对抗性记忆投毒不同，UCC 不需要攻击者；它源于良性交互，这些交互中受限于特定作用域的中间产物被持久化，并在随后被错误应用。我们通过受控评估协议对 UCC 进行了形式化，引入了三种污染类型的分类法，并在两种共享状态机制中评估了该问题。在原始共享状态下，仅良性交互就会产生 57%–71% 的污染率。当共享状态为对话式时，写入时清理是有效的，但当共享状态包含可执行产物时，仍存在显著的残留风险，污染通常表现为隐蔽的错误答案。这些结果表明，共享状态智能体需要文本级清理之外的产物级防御，以防止隐蔽的跨用户失效。

## Abstract
LLM-based agents increasingly operate across repeated sessions, maintaining task states to ensure continuity. In many deployments, a single agent serves multiple users within a team or organization, reusing a shared knowledge layer across user identities. This shared persistence expands the failure surface: information that is locally valid for one user can silently degrade another user's outcome when the agent reapplies it without regard for scope. We refer to this failure mode as unintentional cross-user contamination (UCC). Unlike adversarial memory poisoning, UCC requires no attacker; it arises from benign interactions whose scope-bound artifacts persist and are later misapplied. We formalize UCC through a controlled evaluation protocol, introduce a taxonomy of three contamination types, and evaluate the problem in two shared-state mechanisms. Under raw shared state, benign interactions alone produce contamination rates of 57--71%. A write-time sanitization is effective when shared state is conversational, but leaves substantial residual risk when shared state includes executable artifacts, with contamination often manifesting as silent wrong answers. These results indicate that shared-state agents need artifact-level defenses beyond text-level sanitization to prevent silent cross-user failures.

随着 LLM 智能体（Agents）在团队和组织中的普及，许多智能体被设计为具有“持久化共享状态”，即多个用户共用同一个记忆库或上下文。
*   **研究动机**：过去的研究主要关注恶意攻击者通过投毒手段破坏智能体，但本文发现，即使在**完全良性**的交互中，也会发生“无意间跨用户污染”（Unintentional Cross-User Contamination, UCC）。
*   **核心问题**：用户 A 为了特定任务设定的局部规则（如“去年”指代“过去12个月”），会被智能体错误地当作全局知识存储，并应用到用户 B 的查询中，导致用户 B 得到看似正确实则错误的“静默失败”结果。

## 2. 论文提出的方法论
论文对 UCC 进行了形式化定义，并提出了一套评估与防御框架：
*   **形式化模型**：将智能体操作抽象为写入（Write）、读取（Read）和执行（Agent Function）三个步骤。通过对比“干净状态”和“受污染状态”下智能体的输出差异来衡量污染率。
*   **污染分类法（Taxonomy）**：
    1.  **语义污染 (SC)**：误用特定用户的术语解释（如缩写、阈值定义）。
    2.  **转换污染 (TC)**：误用特定的数据处理规则（如舍入规则、格式化要求）。
    3.  **程序污染 (PC)**：误用特定的工作流或工具调用逻辑（如去重方式、过滤条件）。
*   **防御机制 - 写入时清理 (SSI)**：提出“脱敏共享交互”（Sanitized Shared Interaction）方法。在信息进入共享库之前，利用 LLM 重新编写交互记录，剔除用户特定的偏好和局部约束，仅保留具有通用价值的事实性知识。

## 6. 论文的主要结论与发现
*   **高污染率**：在无防御情况下，良性交互导致的跨用户污染率高达 **57%–71%**。
*   **静默失败风险**：在简单模式下，污染往往导致“错误答案”而非“报错”，这种静默失败对用户来说极难察觉。
*   **防御局限性**：SSI 清理机制在纯文本对话（Slack）中非常有效（污染率降至 6%），但在涉及**可执行代码**（EHRAgent）时效果有限（仍有 33%-41% 的残留风险）。
*   **代码依赖性**：在复杂数据库模式下，智能体倾向于直接复制记忆中的代码模板，导致局部规则通过代码逻辑“硬编码”传播。

## 7. 优点
*   **视角新颖**：首次系统性地探讨了非对抗性、由良性行为引起的跨用户安全风险。
*   **分类严谨**：提出的 SC/TC/PC 分类法为后续研究共享状态智能体的鲁棒性提供了清晰的框架。
*   **现实意义强**：针对当前企业级多用户智能体部署中普遍存在的“知识复用”与“隐私/逻辑隔离”矛盾提出了警示。

## 8. 不足与局限
*   **防御手段单一**：SSI 仅在写入阶段进行文本级处理，无法有效解决嵌入在代码或复杂逻辑产物中的污染。
*   **人工构造偏差**：实验依赖人工设计的公约，可能存在主观性，且未在大规模真实用户日志上进行验证。
*   **架构限制**：研究主要针对基于检索或长上下文的共享机制，对于通过模型微调（Fine-tuning）实现的知识共享是否具有相同特性尚不明确。

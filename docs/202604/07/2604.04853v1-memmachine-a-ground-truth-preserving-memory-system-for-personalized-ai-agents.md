## 摘要
大语言模型（LLM）智能体需要持久化记忆来维持个性化、事实连续性和长程推理，然而标准的上下文窗口和检索增强生成（RAG）流水线在多会话交互中性能会下降。我们提出了 MemMachine，这是一个开源记忆系统，它在保留事实真值的架构中集成了短期记忆、长期情境记忆（episodic memory）和画像记忆（profile memory），该架构存储完整的对话情境并减少了有损的基于 LLM 的提取。MemMachine 使用上下文检索，通过周围上下文扩展核心匹配，从而在相关证据跨越多个对话轮次时提高召回率。在各项基准测试中，MemMachine 实现了强大的准确率-效率权衡：在 LoCoMo 上使用 gpt4.1-mini 达到 0.9169；在 LongMemEvalS (ICLR 2025) 上，六维消融实验达到了 93.0% 的准确率，其中检索阶段的优化——检索深度调整（+4.2%）、上下文格式化（+2.0%）、搜索提示词设计（+1.8%）和查询偏差修正（+1.4%）——优于摄入阶段的增益，如句子分块（+0.8%）。GPT-5-mini 在配合优化提示词时比 GPT-5 高出 2.6%，成为最具成本效益的配置。与 Mem0 相比，在相同条件下，MemMachine 使用的输入 token 减少了约 80%。配套的检索智能体（Retrieval Agent）可以自适应地在直接检索、并行分解或迭代查询链（chain-of-query）策略之间路由查询，在随机噪声条件下，在 HotpotQA-hard 上达到 93.2%，在 WikiMultiHop 上达到 92.6%。这些结果表明，在分层自适应检索的同时保留情境事实真值，可以为个性化 LLM 智能体提供稳健、高效的长期记忆。

## Abstract
Large Language Model (LLM) agents require persistent memory to maintain personalization, factual continuity, and long-horizon reasoning, yet standard context-window and retrieval-augmented generation (RAG) pipelines degrade over multi-session interactions. We present MemMachine, an open-source memory system that integrates short-term, long-term episodic, and profile memory within a ground-truth-preserving architecture that stores entire conversational episodes and reduces lossy LLM-based extraction. MemMachine uses contextualized retrieval that expands nucleus matches with surrounding context, improving recall when relevant evidence spans multiple dialogue turns. Across benchmarks, MemMachine achieves strong accuracy-efficiency tradeoffs: on LoCoMo it reaches 0.9169 using gpt4.1-mini; on LongMemEvalS (ICLR 2025), a six-dimension ablation yields 93.0 percent accuracy, with retrieval-stage optimizations -- retrieval depth tuning (+4.2 percent), context formatting (+2.0 percent), search prompt design (+1.8 percent), and query bias correction (+1.4 percent) -- outperforming ingestion-stage gains such as sentence chunking (+0.8 percent). GPT-5-mini exceeds GPT-5 by 2.6 percent when paired with optimized prompts, making it the most cost-efficient setup. Compared to Mem0, MemMachine uses roughly 80 percent fewer input tokens under matched conditions. A companion Retrieval Agent adaptively routes queries among direct retrieval, parallel decomposition, or iterative chain-of-query strategies, achieving 93.2 percent on HotpotQA-hard and 92.6 percent on WikiMultiHop under randomized-noise conditions. These results show that preserving episodic ground truth while layering adaptive retrieval yields robust, efficient long-term memory for personalized LLM agents.

*   **核心问题**：现有的 LLM 智能体在处理多轮、跨会话的交互时，面临“记忆失效”的问题。传统的 RAG（检索增强生成）主要针对静态文档，而现有的记忆系统（如 Mem0, Zep）过度依赖 LLM 进行事实提取和压缩，导致**操作成本高昂**、**事实提取存在概率性错误**以及**随时间推移产生的误差累积**。
*   **研究背景**：智能体需要像人类一样拥有短期记忆（工作空间）、长期情节记忆（具体经历）和语义记忆（用户画像/事实总结）。MemMachine 的出发点是“保留事实真值”（Ground-Truth-Preserving），即尽可能存储原始对话，而非仅存储 LLM 提取后的摘要。

### 2. 方法论：核心思想与关键技术
*   **核心思想**：采用两层记忆架构（短期 + 长期情节 + 画像记忆），通过存储原始对话情节（Episodes）并结合细粒度的句子级索引，减少对 LLM 提取的依赖。
*   **关键技术细节**：
    *   **情节化存储与索引**：将对话划分为“情节”（Episode），在摄入阶段使用 NLTK 将情节切分为句子并生成嵌入（Embedding），保留原始文本作为事实真值。
    *   **上下文检索（Contextualized Retrieval）**：检索时不只返回最相似的句子，而是以该句子为核心，扩展获取其前后的邻近情节（如前 1 后 2），形成“情节簇”，解决对话数据中语义分散的问题。
    *   **检索智能体（Retrieval Agent）**：引入一个 LLM 编排层，根据查询复杂度将任务路由至三种策略：
        1.  **直接检索**：简单查询。
        2.  **并行分解（SplitQuery）**：针对涉及多个独立实体的查询。
        3.  **迭代查询链（ChainOfQuery）**：针对需要多步推理的复杂依赖链。
    *   **画像记忆（Profile Memory）**：专门提取并维护结构化的用户偏好和稳定属性。

### 6. 主要结论与发现
*   **检索阶段优于摄入阶段**：消融实验显示，检索深度的调整（+4.2%）和上下文格式化（+2.0%）对准确率的提升远大于摄入阶段的句子分块（+0.8%）。
*   **成本效益极高**：MemMachine 在 LoCoMo 上的 Token 消耗比 Mem0 **减少了约 80%**，且准确率更高（0.9169）。

*   **多步推理优势**：Retrieval Agent 在处理复杂依赖链时，通过迭代查询显著提升了召回率（HotpotQA 达到 93.2%）。

### 7. 优点（亮点）
*   **开源与框架无关**：支持 REST API、Python SDK 和 MCP 协议，易于集成。
*   **保留真值**：避免了 LLM 在记忆提取过程中的“幻觉”和信息丢失。
*   **上下文感知**：通过“情节簇”检索，解决了 RAG 在对话场景中因缺乏上下文而导致的理解偏差。
*   **极致的 Token 效率**：通过减少不必要的 LLM 预处理，大幅降低了生产环境的运行成本。

### 8. 不足与局限
*   **时间推理仍有提升空间**：在 LoCoMo 的时间推理维度上，虽然表现优秀，但仍略逊于某些专门针对时间图谱优化的系统（如 Memobase）。
*   **模型依赖性**：系统性能高度依赖于下游 LLM 的指令遵循能力，尤其是 Retrieval Agent 的路由决策。
*   **基础设施要求**：相比于纯文本存储，MemMachine 需要 PostgreSQL 和 Neo4j 等数据库支持，增加了部署的复杂度。
*   **多模态缺失**：目前仅支持文本对话，尚未扩展到图像、音频等跨模态记忆。

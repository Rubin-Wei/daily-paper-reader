## 摘要
训练有效的软件工程智能体需要大量的特定任务轨迹，这导致了高昂的数据构建成本。受数学推理中“少即是多”（Less-Is-More）假设的启发，我们研究了其在智能体场景中的扩展，并提出了一种端到端训练框架，通过更少但更高质量的训练轨迹实现了卓越的智能体能力。这是通过 STITCH（滑动记忆轨迹推理与任务分块启发式）实现的，这是一种由粗到精的机制，通过过滤低价值噪声并保留决策关键的 token，从而最大化训练信号的质量。我们在多个智能体框架（如 mini-SWE-agent、MSWE-agent）、模型规模（30B 到 355B）以及多语言设置（Python、Java 和 ArkTS）上进行了实验。在 SWE-bench Verified 上，使用 STITCH 训练的模型相比基础模型实现了高达 63.16% 的相对提升。在 Multi-SWE-bench (Java) 上，MiniMax-M2.5-STITCH 配合我们的 CodeArts Agent 脚手架达到了 43.75%（+16.67%）。在鸿蒙系统（ArkTS）上，GLM-4.7-STITCH 在不到 1000 条训练轨迹的情况下，将编译通过率提高到 61.31%（+43.34%）。我们的结果证实，“少即是多”范式能够有效地推广到跨不同语言和模型规模的复杂智能体任务中。

## Abstract
Training effective software engineering agents requires large volumes of task-specific trajectories, incurring substantial data construction costs. Inspired by the "Less-Is-More" hypothesis in mathematical reasoning, we investigate its extension to agentic scenarios and propose an end-to-end training framework that achieves superior agentic capabilities with fewer but higher-quality training trajectories. This is achieved via STITCH (Sliding-memory Trajectory Inference and Task Chunking Heuristic), a coarse-to-fine mechanism that filters low-value noise and retains decision-critical tokens to maximize training signal quality. We conduct experiments across multiple agent frameworks (e.g., mini-SWE-agent, MSWE-agent), model scales (30B to 355B), and multilingual settings (Python, Java, and ArkTS). On SWE-bench Verified, models trained with STITCH achieve up to 63.16% relative improvement over base models. On Multi-SWE-bench (Java), MiniMax-M2.5-STITCH achieves 43.75% with our CodeArts Agent scaffold (+16.67%). On HarmonyOS (ArkTS), GLM-4.7-STITCH improves the compilation pass rate to 61.31% (+43.34%) with less than 1K training trajectories. Our results confirm that the "Less-Is-More" paradigm generalizes effectively to complex agentic tasks across diverse languages and model scales.

### 2. 方法论：SandForge 与 STITCH 框架
论文提出了一个端到端的训练管线，包含两个核心组件：
*   **SandForge（数据构建与评估平台）**：
    *   将真实的 GitHub Issue 转化为可执行的任务。
    *   自动验证修复结果（通过测试用例、编译检查等）。
    *   统一不同智能体框架的接口，将执行过程保存为结构化的轨迹（Trajectory）。
*   **STITCH（由粗到精的轨迹筛选机制）**：
    *   **宏观层面（Macro-Level）**：使用轻量级的逻辑回归（LR）模型，基于代码产量、工具使用频率、效率和错误恢复等统计特征，快速过滤掉低质量或失败的轨迹。
    *   **微观层面（Micro-Level）**：针对长轨迹，采用 **Map-Reduce 范式结合滑动记忆机制（Sliding-memory）**。
        *   **启发式安全切分**：在不破坏“动作-观察”逻辑链的前提下进行分块。
        *   **滑动记忆**：在评估当前块时，携带前一块的压缩记忆，确保语义连贯。
        *   **语义提取**：利用 LLM 作为裁判，识别并提取轨迹中高价值的决策片段（即使整体轨迹失败，也能提取局部成功的子任务）。

### 6. 主要结论与发现
*   **“少即是多”有效性**：在所有测试场景中，STITCH 筛选的数据均优于传统的 RFT 方法。
*   **性能提升显著**：
    *   在 SWE-bench Verified 上，30B 模型相对提升达 **63.16%**。
    *   在 Java 场景下，MiniMax-M2.5-STITCH 达到了 **43.75%** 的解决率，刷新了开源模型 SOTA。
    *   在低资源语言 ArkTS 上，仅用不到 1k 条数据就将编译率提升了 **43.34%**。
*   **模型规模效应**：对于 100B 以下的模型，高质量数据的提升效果最为显著；对于 200B 以上的顶尖模型，提升幅度有所收窄，可能接近基准测试的饱和点。

### 7. 优点与亮点
*   **数据效率极高**：证明了智能体能力的激发不需要海量数据，关键在于轨迹的决策质量。
*   **STITCH 机制创新**：通过滑动记忆解决了长轨迹评估中的上下文断裂问题，能够从失败轨迹中“变废为宝”提取高质量片段。
*   **多语言泛化**：不仅限于 Python，在 Java 和新兴的 ArkTS 上也表现优异，证明了方法的通用性。

### 8. 不足与局限
*   **超大模型边际效应**：在 200B+ 的模型上，STITCH 的提升相对较小（约 2-3%），暗示对于极强基础模型，简单的 SFT 可能已不是主要瓶颈。
*   **LLM 裁判依赖**：微观层面的筛选高度依赖于作为裁判的 LLM 的水平，可能存在潜在的评估偏差。
*   **任务复杂度上限**：虽然在 SWE-bench 上表现良好，但对于需要极长工期、跨多个仓库的超大规模软件工程任务，该方法的适用性尚待验证。
